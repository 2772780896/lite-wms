# 大文件解决方案：流式传输架构

当数据处理终端不支持高效并发或聚合的数据体实在过大时，考虑 stream 流式传输。

---

## 1. 核心构想

在单线程的 JavaScript 运行环境中，消灭主线程阻塞，将内存开销从 O(N) 降到 O(1)。

**传统误区**：认为用 `map`/`slice` 加上 `async/await` 就能异步化。实际上，在非连续内存（如普通的 JS 列表/大字符串）上做任何切片，底层全是同步的 C++ 遍历拷贝，严重卡死主线程。

**流式精髓**：不把大任务一口气加载进内存，而是在数据出生（I/O 发生）的那一刻起，就让它以连续、微小的"碎片（Chunk）"形式存在，通过异步的事件驱动，化整为零地处理。

---

## 2. 底层思路

如何做到切片、处理却不触发 JS 层的循环遍历？

- **连续内存（Contiguous Memory）**：必须使用 `ArrayBuffer`、`TypedArray` 或 Node.js 的 `Buffer`。
- **零拷贝切片（Zero-Copy Slicing）**：坚决放弃会触发深拷贝循环的 `slice()`，改用 `subarray()`。它只在原有连续内存上创建新的指针视图（Window），耗时永远是 O(1)，彻底消灭了 JS 层的遍历开销。
- **硬件级搬运（DMA）**：在流的输入输出端（硬盘/网卡），利用操作系统底层的直接内存访问技术（DMA），由硬件芯片直接去连续内存中抓取数据发送。CPU 不参与，JS 主线程零阻塞。

```javascript
// slice() — 深拷贝，O(N)，阻塞主线程
const chunk = buffer.slice(0, 16384);

// subarray() — 零拷贝，O(1)，仅创建视图
const chunk = buffer.subarray(0, 16384);
```

---

## 3. 架构权衡

切片并没有缩短物理传输的"总时间"，那它的架构意义是什么？

- **空间换并发**：将服务器单个请求的内存占用从 O(文件大小) 锁死在 O(1)（如 16KB）。让一台 8GB 内存的服务器，从"只能同时撑起 4 个大下载"蜕变为"能同时挂载 10,000 个下载而不崩溃"。
- **保护主线程（可用性第一）**：利用网络天然的异步延时，把 CPU 从繁重的搬运工作中解放出来。在微观的间隙中，Event Loop 可以去处理成千上万个其他用户的并发请求。
- **高级网络红利**：由于数据天生是按块（Chunk）可控的，天然支持断点续传、前端暂停/重连，以及防止网络丢包的背压（Backpressure）机制。

---

## 4. 可行性评估

完全可行，且是行业标配：Node.js 的 `fs.createReadStream`、Web 端的 Streams API、以及大文件分片上传/大吞吐量数据清洗系统，底层全是在玩这套指针和连续内存的游戏。

**适用边界：**

| 场景 | 可行性 | 原因 |
|------|--------|------|
| 普通 JS 业务列表（Array） | 不可行 | 底层是非连续内存，无法用 `subarray`，必须通过 `worker_threads` 丢到后台遍历 |
| 二进制/文件/网卡字节流 | 极度完美可行 | 连续内存 + 零拷贝切片 + DMA，天然适配 |
