# Node.js 并发模型学习笔记

## 1. 核心误区澄清

> Node.js 是"单线程"的，并不意味着它不能处理高并发。

| 误解 | 真相 |
| --- | --- |
| Node.js 是单线程，所以不能高并发 | 错。Node.js 运行时是多线程的，单线程仅指 V8 执行 JS 代码的线程。 |
| `await` 会阻塞主线程 | 错。`await` 是让出执行权，主线程立刻去处理下一个请求。 |
| 数据库操作是 JS 算的 | 错。数据库是独立的多线程程序，Node.js 只是发号施令。 |

## 2. 餐厅老板模型（核心隐喻）

- **老板 (V8 引擎)**：单线程。只负责接单、算账、发指令。绝不干重活。
- **后厨 (Libuv 线程池)**：默认 4 个工人。负责切菜（文件 I/O、Crypto、DNS）。
- **中央厨房 (数据库/OS)**：多核多线程。负责核心业务（库存扣减、复杂查询）。
- **传菜口 (Event Loop)**：负责把做好的菜（回调/Promise）端给老板。

**协作流程**：用户 A 请求接口，Node 把"查询数据库"这个任务丢给后厨，然后立刻去接待用户 B。后厨做完了，按铃（Callback/Promise），老板再把结果返回给用户 A。

**结论**：Node.js 的高并发能力来自于"我不等你，我先做别的事"。因此，在库存系统中，绝不能在主线程里用 `for` 循环去等待每一个库存扣减的结果，必须交给数据库去并行处理。

## 3. 事件循环与任务队列

### 3.1 宏任务 (Macrotask)

- **包含**：`setTimeout`、`setInterval`、I/O 回调、整体 Script 代码。
- **特点**：一个事件循环阶段只执行一个宏任务。

### 3.2 微任务 (Microtask)

- **包含**：`Promise.then/catch/finally`、`process.nextTick`、`await` 之后的代码。
- **特点**：执行优先级高于宏任务。当一个宏任务执行完后，会清空所有微任务，再执行下一个宏任务。

### 3.3 `await` 的本质

```javascript
console.log('1');
await db.query();  // 老板喊：去查库！（立刻转身去接新客）
console.log('2');  // 这是微任务，等库查完且主线程空闲时执行
```

**结论**：`await` 让老板（主线程）空闲下来，没有阻塞。

## 4. Worker Threads 与多核

### 4.1 为什么需要它？

Libuv 线程池是为 Node.js 内部 I/O 服务的。如果你有一段纯 JS 的死循环（如大数计算），Libuv 帮不了你，会卡死老板。

### 4.2 它是怎么工作的？

1. **克隆 V8**：Node.js 会在内存中创建一个新的 V8 实例（一个新的"老板"）。
2. **操作系统分配**：操作系统将两个 V8 实例分配给不同的 CPU 核心。
3. **隔离**：两个线程不共享内存，通过 `postMessage` 通信（像两个独立的餐厅）。

### 4.3 与单线程的关系

- **JS 语言规范**：单线程（一个 V8 实例内）。
- **Node.js 运行时**：多线程（通过 Worker Threads 创建多个 V8 实例）。

**结论**：不矛盾。这是通过"隔离"实现的并行，避免了传统多线程的锁问题。

## 5. 为什么数据库能处理高并发（而 Node 不行）？

| 特性 | Node.js（应用层） | Database（数据层） |
| --- | --- | --- |
| **主要角色** | 业务逻辑编排、数据搬运工 | 数据存储、计算、守门员 |
| **并发机制** | 单线程 + 异步 I/O（适合 I/O 密集型） | 多线程/多进程（真正的同时干活） |
| **数据一致性** | 弱（内存中的变量容易被多个请求改乱） | 强（通过锁 Lock 和事务 Transaction） |
| **处理方式** | 代码逻辑：`if (stock > 0) { deduct(); }` | 物理机制：`UPDATE inventory SET qty = qty - 1 WHERE qty > 0` |

核心原因在于三点：

- **行级锁 (Row-Level Locking)**：执行 `UPDATE inventory WHERE id = 1` 时，数据库会暂时锁住这一行数据，其他请求必须排队，防止"超卖"。
- **原子性 (Atomicity)**：数据库的一条 SQL 指令是不可分割的最小单位，要么执行成功，要么失败回滚。
- **连接池 (Connection Pool)**：数据库专门为并发连接做了优化，可以同时处理成千上万个连接请求。

## 6. 库存系统实战策略

### 6.1 并发安全（防超卖）

> **原则**：绝不先查再改。

```javascript
// ❌ 错误：先查（10个）→ 再改（10-1），并发下会变成 9
const stock = await prisma.inventory.findUnique(...);
if (stock > 0) await prisma.update(...);

// ✅ 正确：一条命令搞定（原子操作）
await prisma.inventory.updateMany({
  where: { id: 1, quantity: { gte: 1 } }, // 数据库层面判断
  data: { quantity: { decrement: 1 } }    // 数据库层面扣减
});
```

### 6.2 批量处理：打包、传输与执行

针对"同时出库多个商品"的场景，整个流程分为三个阶段：

**第一阶段：JS 层（打包）** — 使用 `Array.map()` 遍历前端数据，将其转换成 Prisma/SQL 需要的数组格式。这一步只是组装数据，没有触碰数据库。

```javascript
const data = items.map(item => ({
  productId: item.id,
  quantity: item.count
}));
```

**第二阶段：ORM 层（传输）** — 调用 `prisma.createMany()` 或 `updateMany()`，ORM 将 JS 对象转换成一条 SQL 语句发送给数据库。

**第三阶段：数据库层（执行）** — 数据库接收到一条 `INSERT` 或 `UPDATE` 指令，引擎内部多线程高效处理这批数据，并通过事务锁保证一致性。

**设计优势**：只发了 1 个 HTTP 请求、1 条 SQL 指令，却处理了 N 条数据，极大减轻了网络带宽和数据库连接的负担。

### 6.3 事务一致性

- **场景**：扣库存 + 记流水。
- **方案**：`prisma.$transaction(...)`。
- **结果**：要么全成功，要么全失败（回滚）。

## 7. 什么时候用 Worker Threads？

| 场景 | 方案 | 原因 |
| --- | --- | --- |
| 库存出库 | 主线程 + 数据库 | I/O 密集型，Node.js 天生擅长。 |
| 密码哈希 | Libuv 线程池 (`crypto`) | Node.js 内部已封装。 |
| 生成年度报表 | Worker Threads | CPU 密集型，需隔离计算。 |
| 复杂图像/视频处理 | Worker Threads | CPU 密集型。 |

## 8. 一句话总结

Node.js 利用"单线程 JS + 多线程 I/O + 多进程/线程扩展"的架构，将并发控制下沉到数据库，从而在保证数据一致性的前提下，实现高吞吐。

>  **具体实现**
>
> 在设计中，Node.js 只负责编排流程和参数校验，而将并发控制和数据一致性下沉到数据库层，利用 Prisma 的原子操作和事务机制，避免 Node.js 单线程在处理复杂计算时的阻塞风险。
