# 轻量级智能仓储管理系统 (WMS Lite)

## 1. 设计思路与选型

### 1.1 技术选型

- **后端框架**：Node.js + Express/Koa（或 NestJS）
- **ORM**：Prisma（选择原因：类型安全，Schema 清晰，内置事务支持，提高开发效率）
- **数据库**：SQLite（开发环境）/ PostgreSQL（生产环境，支持更高并发）
- **前端**：React + Ant Design（用于管理后台可视化）

### 1.2 核心设计原则

- **业务解耦**：商品、仓库、库存流水分离，避免数据冗余。
- **并发安全（重点）**：摒弃"先查再改"的非原子模式，采用数据库层面的原子更新，防止超卖。
- **可追溯性**：所有出入库操作必须留存流水记录（Transaction Log）。

## 2. 数据库设计 (Prisma Schema)

```prisma
// schema.prisma

model Product {
  id        Int      @id @default(autoincrement())
  sku       String   @unique // 商品编码
  name      String
  spec      String?
  inventory Inventory[] // 关联库存
}

model Warehouse {
  id        Int      @id @default(autoincrement())
  name      String
  address   String?
  inventory Inventory[] // 关联库存
}

model Inventory {
  id           Int      @id @default(autoincrement())
  productId    Int
  warehouseId  Int
  quantity     Int      @default(0) // 可用库存
  lockedQty    Int      @default(0) // 锁定库存（预留字段，用于预扣）
  product      Product  @relation(fields: [productId], references: [id])
  warehouse    Warehouse @relation(fields: [warehouseId], references: [id])

  @@unique([productId, warehouseId]) // 唯一约束，防止重复建仓
}

model TransactionLog {
  id          Int      @id @default(autoincrement())
  bizNo       String   @unique // 业务单号 (UUID)，用于幂等性
  productId   Int
  warehouseId Int
  type        String   // IN / OUT
  quantity    Int
  operatorId  Int
  createdAt   DateTime @default(now())
}
```

## 3. 核心业务逻辑实现 (Node.js + Prisma)

### 3.1 销售出库（高并发安全版）

> 设计亮点：使用 `updateMany` 结合 `where` 条件，实现原子扣减。

```javascript
/**
 * 处理销售出库
 * @param {number} productId - 商品ID
 * @param {number} warehouseId - 仓库ID
 * @param {number} count - 出库数量
 * @param {string} bizNo - 业务单号（前端生成 UUID，防止重复提交）
 */
async function handleOutbound(productId, warehouseId, count, bizNo) {
  // 1. 开启事务，确保"扣库存"和"记流水"同时成功或失败
  return await prisma.$transaction(async (tx) => {

    // 2. 原子扣减库存（核心）
    // 关键：WHERE 条件中直接判断 quantity >= count
    const updateResult = await tx.inventory.updateMany({
      where: {
        productId: productId,
        warehouseId: warehouseId,
        quantity: { gte: count }, // 数据库层面的库存校验
      },
      data: {
        quantity: { decrement: count }, // 原子操作，数据库内部执行减法
      },
    });

    // 3. 校验扣减结果
    if (updateResult.count === 0) {
      throw new Error('库存不足或商品不存在');
    }

    // 4. 记录流水（幂等性设计：bizNo 唯一索引防止重复插入）
    await tx.transactionLog.create({
      data: {
        bizNo: bizNo,
        productId: productId,
        warehouseId: warehouseId,
        type: 'OUT',
        quantity: count,
      },
    });

    return { success: true, message: '出库成功' };
  });
}
```

### 3.2 批量入库（高性能版）

> 设计亮点：利用 Prisma 的 `createMany` 一次性写入，减少数据库 I/O。

```javascript
async function handleBatchInbound(items) {
  // items = [{productId: 1, warehouseId: 1, count: 10}, ...]
  const data = items.map(item => ({
    productId: item.productId,
    warehouseId: item.warehouseId,
    quantity: item.count
  }));

  // 直接交给数据库，让数据库引擎处理批量插入
  await prisma.inventory_transactions.createMany({
    data: data,
    skipDuplicates: true, // 跳过重复主键
  });
}
```

## 4. 权限设计 (RBAC 简化版)

| 角色 | 权限标识 | 说明 |
| --- | --- | --- |
| 管理员 (Admin) | `system:admin` | 拥有所有路由权限，管理商品和仓库 |
| 仓库员 (Staff) | `inventory:operate` | 仅允许访问 `/api/inbound` 和 `/api/outbound` |

实现方式：Express 中间件校验 Token 中的 Role 字段。

## 5. 高并发应对策略（面试必问）

| 场景 | 解决方案 | 说明 |
| --- | --- | --- |
| 超卖问题 | 原子更新 (Atomic Update) | 使用 `updateMany` 配合 `quantity: { gte: count }`，杜绝"先查再改"的竞态条件。 |
| 重复提交 | 业务单号幂等 | 流水表 `bizNo` 设置唯一索引，即使前端重复发请求，数据库也会拦截。 |
| 性能瓶颈 | 批量操作 & 异步化 | 批量入库用 `createMany`；非核心逻辑（如发邮件、统计报表）放入消息队列（未来扩展）。 |
| 数据一致性 | 数据库事务 (`$transaction`) | 确保库存扣减和流水记录在同一个事务中，要么全成，要么全败。 |

## 6. 总结与反思

- **为什么不用 Redis/MQ？** 针对当前业务体量（中小企业），数据库事务足以支撑，避免过度设计。
- **前端视角**：作为全栈开发者，后端接口设计遵循 RESTful 规范，便于前端 React 进行状态管理（如 React Query 做缓存失效）。
