# 核心技术点总结 (Key Technical Summary)

## 1. JavaScript 同步性与 Node.js 并发的关系

> **误区澄清**：Node.js 是"单线程"的，并不意味着它不能处理高并发。

- **JS 主线程（老板）**：只有一个。它只负责执行你的业务逻辑（路由分发、参数校验、调用 ORM）。它是同步、阻塞的。
- **Libuv 线程池 & 操作系统（后厨）**：Node.js 底层有额外的线程专门处理 I/O 操作（数据库查询、文件读写、网络请求）。
- **协作模式（事件循环）**：
  1. 用户 A 请求接口，Node 把"查询数据库"这个任务丢给后厨，然后立刻去接待用户 B。
  2. 后厨做完了，按铃（Callback/Promise），老板再把结果返回给用户 A。

**结论**：Node.js 的高并发能力来自于"我不等你，我先做别的事"。因此，在我们的库存系统中，绝不能在主线程里用 `for` 循环去等待每一个库存扣减的结果，必须交给数据库去并行处理。

## 2. 多 ID 入库/出库：打包、拆包与执行流程

针对"同时出库多个商品"的场景，流程如下：

### 第一阶段：JS 层（打包 - Packing）

- **动作**：使用 `Array.map()` 遍历前端传来的 ID 列表。
- **目的**：数据清洗与格式化。将杂乱的前端数据转换成 Prisma/SQL 需要的数组格式。
- **代码体现**：

```javascript
// 打包：把对象数组转换成纯数据数组
const data = items.map(item => ({
  productId: item.id,
  quantity: item.count
}));
```

> **注意**：这里只是组装数据，没有触碰数据库。

### 第二阶段：ORM 层（传输 - Transmitting）

- **动作**：调用 `prisma.createMany()` 或 `updateMany()`。
- **目的**：发送指令。ORM 会将 JS 对象转换成一条 SQL 语句。

### 第三阶段：数据库层（拆包与执行 - Execution）

- **动作**：数据库接收到一条 `INSERT` 或 `UPDATE` 指令。
- **目的**：原子执行。数据库引擎（如 PostgreSQL/MySQL）内部是多线程的，它会高效地处理这一批数据，并且通过事务锁保证这批数据的一致性。

**设计优势**：我们只发了 1 个 HTTP 请求，1 条 SQL 指令，却处理了 N 条数据，极大减轻了网络带宽和数据库连接的负担。

## 3. 为什么数据库能处理高并发（而 Node 不行）？

| 特性 | Node.js（应用层） | Database（数据层） |
| --- | --- | --- |
| **主要角色** | 业务逻辑编排、数据搬运工 | 数据存储、计算、守门员 |
| **并发机制** | 单线程 + 异步 I/O（适合 I/O 密集型） | 多线程/多进程（真正的同时干活） |
| **数据一致性** | 弱（内存中的变量容易被多个请求改乱） | 强（通过锁 Lock 和事务 Transaction） |
| **处理方式** | 代码逻辑：`if (stock > 0) { deduct(); }` | 物理机制：`UPDATE inventory SET qty = qty - 1 WHERE qty > 0` |

**核心原因**：

- **行级锁 (Row-Level Locking)**：当我们执行 `UPDATE inventory WHERE id = 1` 时，数据库会暂时锁住这一行数据，其他请求必须排队，防止"超卖"。
- **原子性 (Atomicity)**：数据库的一条 SQL 指令是不可分割的最小单位。要么执行成功，要么失败回滚。
- **连接池 (Connection Pool)**：数据库专门为并发连接做了优化，可以同时处理成千上万个连接请求。

> 💡 **给面试官的"一句话总结"**
>
> "在我的设计中，Node.js 只负责编排流程和参数校验，而将并发控制和数据一致性下沉到数据库层，利用 Prisma 的原子操作和事务机制，避免了 Node.js 单线程在处理复杂计算时的阻塞风险。"
