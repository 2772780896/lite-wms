# WMS Lite 仓储管理粒度与并发方案说明

## 一、管理粒度：SKU 与 SN

在 WMS 设计中，库存管理的最小粒度分为两个层级：

### SKU（Stock Keeping Unit — 库存量单位）

- **定义**：规格层面的最小粒度。只要颜色、尺寸、型号有任何不同，就是不同的 SKU。
- **示例**：`iPhone-16-黑色-128G` 和 `iPhone-16-白色-128G` 是两个不同的 SKU。
- **管理方式**：只关心"某种商品有多少件"，不关心具体是哪几件。

### SN（Serial Number — 唯一序列号）

- **定义**：物理实体层面的最小粒度（一品一码）。每件商品拥有独一无二的身份证。
- **示例**：手机 IMEI 码、奢侈品防伪码。同一 SKU 下的每一件都有独立编号。
- **管理方式**：精确追踪"哪一件在哪个仓库、什么状态"。

### 粒度对比

| | 数量级管理（本项目采用） | 单品级管理（本项目不采用） |
|--|--|--|
| **核心思路** | 一个 SKU = 一种商品，库存只记录数量 | 每一件实物都有唯一序列号 |
| **入库** | SKU + 仓库编码 + 数量 → 库存 +N | 逐件扫描/录入序列号 |
| **出库** | SKU + 仓库编码 + 数量 → 库存 -N | 指定具体哪些序列号出库 |
| **适用场景** | 快消品、标品（矿泉水、手机壳、螺丝） | 高价值商品（手机、电脑、奢侈品） |
| **操作成本** | 低（输入数字即可） | 高（逐件扫码/录入） |

### 本项目的选择理由

1. **需求匹配**：需求描述的是"某公司产品"、"多个仓库"、"采购入库/销售出库"，是典型的标品仓储场景。
2. **复杂度控制**：单品级管理意味着入库 500 件要生成/录入 500 个序列号，出库 200 件要逐一指定，对轻量级 WMS 来说操作成本过高。
3. **YAGNI 原则**：当前业务不需要追踪到单品，不需要的功能不加。

### 未来扩展路径

在现有 5 张表基础上，新增一张 **Item（单品表）**：

| 字段 | 说明 |
|------|------|
| serialNo（唯一） | 单品序列号，入库时生成或录入 |
| productId | 关联商品表，属于哪个 SKU |
| warehouseId | 当前所在仓库 |
| status | 在库 / 已出库 |

业务逻辑变化：

- **入库**：库存表 +N 的同时，创建 N 条 Item 记录。
- **出库**：库存表 -N 的同时，将指定 N 条 Item 的 status 改为"已出库"。
- **库存查询**：除了看数量，还能看具体有哪些序列号在库。

现有表结构无需改动，只需新增 Item 表并在入出库逻辑中增加对 Item 的操作。

## 二、高并发防超卖：边界条件 + 数据库隐式行锁

无论数量级还是单品级管理，防超卖的核心思路一致：**把安全校验条件写在 UPDATE 的 WHERE 子句中，让数据库帮你守住底线，而不是在应用层"先查再改"。**

### 为什么不能用"先查再改"

```javascript
// ❌ 错误示范：存在竞态条件
const item = await db.find({ sku: 'A' });  // 查到 quantity = 10
// 此时另一个并发请求也查到了 quantity = 10
if (item.quantity >= 5) {
  await db.update({ sku: 'A' }, { quantity: item.quantity - 5 });  // 两个请求都写入 5
}
// 结果：库存变成了 5，实际应该是 0 → 超卖了 5 件
```

### 模式 A：数量级管理 — WHERE 边界值扣减

出库时，在 WHERE 中直接约束 `quantity >= N`，数据库的行锁保证并发请求排队执行：

```javascript
const result = await prisma.inventory.updateMany({
  where: {
    productId: 1,
    warehouseId: 1,
    quantity: { gte: count },  // 底线：库存必须够
  },
  data: {
    quantity: { decrement: count },
  },
});

if (result.count === 0) {
  return { success: false, message: '库存不足' };
}
```

**原理**：不需要手动写 `FOR UPDATE` 或锁表。数据库底层的隐式行锁让并发请求排队，当库存扣到不足时，`quantity >= count` 不满足 → `affectedRows = 0` → 安全拒绝。

### 模式 B：单品级管理 — WHERE 状态机变更

出库时，在 WHERE 中约束 `status = '在库'`，即使两个仓管员同一毫秒扫了同一个 SN，行锁也只让一个人成功：

```javascript
const result = await db.query(
  `UPDATE wms_sn SET status = '已出库' WHERE sn_code = ? AND status = '在库'`,
  [snCode]
);

if (result.affectedRows === 0) {
  return { success: false, message: '该商品已被出库' };
}
```

### 两种模式对比

| | 数量级（模式 A） | 单品级（模式 B） |
|--|--|--|
| **守什么** | 数量底线 | 状态底线 |
| **WHERE 条件** | `quantity >= N` | `status = '在库'` |
| **防超卖原理** | 库存不够就拒绝 | 已出库/被锁定就拒绝 |
| **并发安全** | 行锁排队，先到先得 | 行锁排队，先到先得 |

## 三、单品级管理的库存统计思维转换

如果未来引入 SN 单品级管理，有一个关键的**思维转换**：

### 前端思维 vs 后端思维

| | 前端思维（State-driven） | 后端思维（Pull on demand） |
|--|--|--|
| **习惯** | 数据变了就立刻更新 State、重新渲染 | 数据存在数据库里，有人要看的时候再查 |
| **误区** | "库存数字必须实时维护在某个字段里" | 库存总数可以在查询时动态聚合 |
| **代价** | 维护"总数字段"的锁代价极高 | 利用索引临时 COUNT 一下，代价极低 |

**核心认知**：数据库不是监控显示器，没有人 24 小时盯着表看。只要底层明细（SN 记录）是绝对准确的，宏观总数永远不会错。**"算"永远比"维护"便宜。**

### 单品级库存统计方式

不维护独立的"库存数量字段"，而是从 SN 明细表动态聚合：

```javascript
const count = await prisma.wms_sn.count({
  where: {
    sku_code: 'iPhone-16-黑-128G',
    warehouseId: 1,
    status: '在库',
  },
});
```

依靠 `(sku_code, status)` 上的复合索引（B+ 树），COUNT 查询只扫索引不扫数据行，速度极快。

## 四、单品级管理的三种架构方案（未来扩展参考）

### 方案一：纯 SN 明细统计法（推荐，适合本项目扩展）

- **原理**：取消独立的库存数量表，库存总数完全靠对 SN 明细表实时 COUNT。
- **出入库**：直接用 `updateMany` + 状态边界条件，不争抢行锁。
- **查总数**：走复合索引聚合。
- **优点**：架构极简，天然免疫"两表数据不对账"的问题。
- **缺点**：单表数据量突破千万级时 COUNT 会有性能瓶颈。
- **适用**：中小型仓库、高价值商品仓。

### 方案二：双表批量增量法

- **原理**：同时保留 SN 明细表和库存数量表。
- **关键约束**：绝对不能每扫一个 SN 就去事务里 +1（行锁竞争）。必须批量提交：扫完一整批后，一次性 INSERT N 条 SN + 一次性 UPDATE 数量 +N。
- **优点**：查库存总数时直接读现成数字，极致快。
- **缺点**：必须用强事务保证双表原子化，代码 Bug 容易导致对账不一致。
- **适用**：大型物流中心、SKU 繁多且高频批量进出。

### 方案三：缓存异步扣减法

- **原理**：Redis 做前锋（内存原子计数），数据库做后卫（异步落库），消息队列削峰。
- **优点**：抗并发能力高出几个数量级。
- **缺点**：架构极复杂，需要 Redis + MQ + 最终一致性对账。
- **适用**：双11/黑五级别的超高并发供应链。

### 方案对比

| 维度 | 方案一：纯 SN 统计 | 方案二：双表批量 | 方案三：Redis 异步 |
|------|------|------|------|
| 开发难度 | 简单（单表） | 中等（双表+批量事务） | 极高（多中间件） |
| 核心依赖 | 数据库索引 + 隐式锁 | 数据库强事务 + 批量包装 | Redis 原子计数 + MQ |

## 五、单品级批量出库的最佳实践（未来扩展参考）

如果引入 SN 管理，批量出库应采用**状态机 Diff 方案**而非传统全局事务回滚：

### 流程

1. **前端批量打包提交** — 扫码枪连续录入一批 SN，一个 POST 请求提交。
2. **后端盲打批量更新** — `updateMany` + `IN` + 边界条件（`status = '在库'`），符合条件的改掉，不符合的自动跳过。
3. **反向捞取成功者** — 从提交列表中查出状态确实变为"已出库"的记录。
4. **内存求差集** — 原始提交列表 vs 成功列表，差集就是失败的（被他人抢先出库的）。
5. **精准响应** — 成功的返回成功，失败的返回失败，前端分别展示。

### 与传统事务回滚的对比

| 维度 | 传统全局事务（全成功或全回滚） | 状态机 Diff（部分成功） |
|------|------|------|
| 并发抗压 | 差（多人同货位易死锁） | 极强（支持海量扫码枪并发） |
| 现场容错 | 低（错一个全部重新扫） | 高（成功的锁定，失败的前端标红替换） |
| 对账风险 | 零风险 | 需依赖流水日志审计 |
