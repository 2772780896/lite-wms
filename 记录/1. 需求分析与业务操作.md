# WMS Lite 业务操作汇总

## 1. 认证

- **登录**：用户输入用户名和密码，系统验证后签发 JWT（含 userId、role），前端存储 Token 用于后续请求。

> **→ 需要：用户表**
> - 用户名（唯一）、密码（哈希）、显示名称、角色（管理员/仓库人员）

---

## 2. 商品管理（仅管理员）

- **新增商品**：输入 SKU 编码、商品名称、规格（可选，用于区分同类商品的不同属性，如"500ml"、"红色/XL码"、"304不锈钢"——同一商品名可能因规格不同而对应不同 SKU，仓库人员需要通过规格快速识别具体是哪个版本的商品），系统创建商品记录。
- **编辑商品**：根据商品 ID 修改名称、规格等信息。
- **查询商品列表**：支持按名称或 SKU 模糊搜索，返回商品列表。
- **查询商品详情**：根据商品 ID 返回单个商品信息。

> **→ 需要：商品表**
> - 商品编码 SKU（唯一，用于扫码反查）、名称、规格（可选）

---

## 3. 仓库管理（仅管理员可增改，所有人可查）

- **新增仓库**：输入仓库编码、名称、地址（可选），系统创建仓库记录。
- **编辑仓库**：根据仓库 ID 修改名称、地址等信息。
- **查询仓库列表**：返回所有仓库信息。
- **查询仓库详情**：根据仓库 ID 返回单个仓库信息。

> **→ 需要：仓库表**
> - 仓库编码（唯一，用于扫码反查）、名称、地址（可选）

---

## 4. 采购入库（管理员 + 仓库人员）

- **单品入库**：操作员输入/扫描商品 SKU + 仓库编码 + 数量，系统根据编码反查商品和仓库，在事务中执行"库存增加 + 记录流水"。如果该商品在该仓库还没有库存记录，则自动创建（通过库存表的 `@@unique([productId, warehouseId])` 联合唯一约束 + Prisma 的 `upsert` 实现：数据库层面单条 SQL 完成"有则加、无则建"，不存在"先查再改"的竞态条件，避免并发入库时创建出重复的库存记录）。
- **批量入库**：一次提交多个商品的入库信息，在同一个事务中批量处理。

> **→ 需要：库存表**
> - 商品 ID + 仓库 ID（联合唯一）、可用数量
>
> **→ 需要：流水表**（入库/出库/退货共用）
> - 业务单号（唯一，幂等性）、商品 ID、仓库 ID、类型（入库/出库/退货入库/退货出库）、数量、操作人 ID、备注（退货原因等）、操作时间
>
> **→ 关联**：通过 SKU 反查商品表，通过仓库编码反查仓库表

---

## 5. 销售出库（管理员 + 仓库人员）

- **单品出库**：操作员输入/扫描商品 SKU + 仓库编码 + 数量，系统根据编码反查商品和仓库，原子校验库存是否充足，在事务中执行"库存扣减 + 记录流水"。库存不足则拒绝操作。

> **→ 复用：库存表**（原子扣减，WHERE quantity >= count）
> **→ 复用：流水表**（记一笔 OUT 类型记录）
>
> **关键约束**：库存表的 quantity 字段需要支持数据库层面的原子更新（updateMany + gte 条件）

---

## 6. 退货（管理员 + 仓库人员）

- **销售退货（退货入库）**：客户退回商品，操作员输入 SKU + 仓库编码 + 数量 + 退货原因，库存增加，流水类型标记为 RETURN_IN，备注退货原因。
- **采购退货（退货出库）**：退回供应商，操作员输入 SKU + 仓库编码 + 数量 + 退货原因，库存扣减，流水类型标记为 RETURN_OUT，备注退货原因。

> **→ 复用：库存表**（增或减）
> **→ 复用：流水表**（类型为 RETURN_IN / RETURN_OUT，备注字段记录退货原因）
>
> **补充**：流水表的"备注"字段在退货场景下为必填

---

## 7. 库存查询（管理员 + 仓库人员）

- **按仓库查库存**：查看某个仓库下所有商品及数量。
- **按商品查库存**：查看某个商品在各仓库的库存分布。
- **库存汇总**：按商品汇总总库存（跨仓库求和），用于管理员看板。

> **→ 复用：库存表**，需要关联查询商品表（名称、SKU）和仓库表（名称、编码）
>
> **查询需求**：支持按 warehouseId 筛选、按 productId 筛选、按 productId 分组求和

---

## 8. 流水查询（管理员 + 仓库人员）

- **查询流水记录**：支持按商品、仓库、时间范围、流水类型筛选，返回出入库操作历史。

> **→ 复用：流水表**
>
> **查询需求**：支持按 productId、warehouseId、type、createdAt 范围筛选
>
> **索引需求**：商品+仓库 组合索引、时间索引

---

## 管理粒度说明：数量级管理 vs 单品级管理

在仓储系统中，"编码入库出库"存在两种不同的管理粒度：

### 数量级管理（本项目采用）

- **核心思路**：一个 SKU = 一种商品，库存只记录数量，不区分每一件实物。
- **入库操作**：输入 SKU + 仓库编码 + 数量 → 库存 +N
- **出库操作**：输入 SKU + 仓库编码 + 数量 → 库存 -N
- **适用场景**：快消品、标品（矿泉水、手机壳、螺丝等），同 SKU 的每一件商品没有区别。
- **示例**：上海仓有 500 件 TSHIRT-RED-S，出库 10 件，剩余 490 件。不关心出库的到底是"哪 10 件"。

### 单品级管理（本项目不采用，但需了解）

- **核心思路**：每一件实物都有唯一序列号，入库/出库精确到具体某一件。
- **入库操作**：逐件扫描/录入序列号 → 每件商品单独记录入库
- **出库操作**：指定具体哪些序列号出库 → 逐一标记状态变更
- **适用场景**：高价值商品（手机、电脑、奢侈品、医疗器械），需要追踪每件商品的流向、保修、召回。
- **示例**：上海仓有 iPhone-15 序列号 001~500，出库时指定序列号 003、007、012。

### 为什么当前只做数量级管理

1. **需求匹配**：需求描述的是"某公司产品"、"多个仓库"、"采购入库/销售出库"，是典型的标品仓储场景。
2. **复杂度控制**：单品级管理意味着入库 500 件要生成/录入 500 个序列号，出库 200 件要逐一指定，对轻量级 WMS 来说操作成本过高。
3. **YAGNI 原则**：当前业务不需要追踪到单品，不需要的功能不加。

### 如果未来要扩展到单品级管理，怎么做

在现有 5 张表基础上，新增一张 **Item（单品表）**：

| 字段 | 说明 |
|------|------|
| serialNo（唯一） | 单品序列号，入库时生成或录入 |
| productId | 关联商品表，表示这个单品属于哪个 SKU |
| warehouseId | 当前所在仓库 |
| status | 在库 / 已出库 |
| createdAt | 入库时间 |

业务逻辑变化：

- **入库**：在库存表 +N 的同时，创建 N 条 Item 记录（序列号自动生成或逐件录入）。
- **出库**：在库存表 -N 的同时，将指定 N 条 Item 的 status 改为"已出库"。
- **库存查询**：除了看数量，还能看具体有哪些序列号在库。

现有表结构无需改动，只需新增 Item 表并在入库/出库逻辑中增加对 Item 的操作。

## 推导总结

从以上业务操作逐步推导，共需要 **5 张表**：

| 表 | 推导来源 | 核心字段 |
|----|----------|----------|
| **用户表** | 认证 | 用户名、密码、名称、角色 |
| **商品表** | 商品管理 | SKU（唯一）、名称、规格 |
| **仓库表** | 仓库管理 | 编码（唯一）、名称、地址 |
| **库存表** | 入库、出库、退货、库存查询 | 商品ID + 仓库ID（联合唯一）、数量 |
| **流水表** | 入库、出库、退货、流水查询 | 业务单号（唯一）、商品ID、仓库ID、类型、数量、操作人ID、备注、时间 |
