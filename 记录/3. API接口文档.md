# WMS Lite API 接口文档

> 所有接口统一前缀 `/api`，响应格式统一为 `{ code, data, message }`。
> 需登录接口在 Header 中携带 `Authorization: Bearer <token>`。

## 通用参数

所有列表接口统一支持以下参数：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| page | number | 1 | 页码 |
| pageSize | number | 20 | 每页条数 |
| sortBy | string | createdAt | 排序字段（各接口允许字段不同，见下表） |
| sortOrder | string | desc | 排序方向：`asc` / `desc` |

**各接口允许的 sortBy 字段：**

| 接口 | 允许字段 |
|------|----------|
| GET /products | createdAt, name, sku |
| GET /warehouses | createdAt, name, code |
| GET /inventory（列表模式） | createdAt, quantity |
| GET /transactions | createdAt, quantity |

## 认证

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/api/auth/login` | Public | 登录 |

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| username | string | 是 | 用户名 |
| password | string | 是 | 密码（明文传输，生产环境需 HTTPS） |

**响应：**

| 字段 | 说明 |
|------|------|
| data.token | JWT token，后续请求放在 Authorization Header |
| data.role | 用户角色（`ADMIN` / `STAFF`），明文返回供前端控制 UI |

**错误码：**

| code | 说明 |
|------|------|
| 400 | 参数缺失 |
| 401 | 用户名或密码错误 |

## 商品管理

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/api/products` | ADMIN | 新增商品 |
| PUT | `/api/products/:sku` | ADMIN | 编辑商品 |
| GET | `/api/products` | ALL | 商品列表（支持模糊搜索 + 分页） |
| GET | `/api/products/:sku` | ALL | 商品详情（纯主数据，库存走 `/inventory?sku=X`） |

> ALL 可查：仓库人员入库/出库时需要通过下拉框选择商品。

**请求参数：**

| 接口 | 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| POST /products | sku | string | 是 | 商品编码（唯一） |
| | name | string | 是 | 商品名称 |
| | spec | string | 否 | 规格描述 |
| PUT /products/:sku | name | string | 否 | 商品名称 |
| | spec | string | 否 | 规格描述 |
| GET /products | keyword | string | 否 | 模糊搜索（匹配 name 或 sku） |
| | page | number | 否 | 页码，默认 1 |
| | pageSize | number | 否 | 每页条数，默认 20 |

**响应示例：**

POST /products、PUT /products/:sku、GET /products/:sku 返回 `data`：

```json
{
  "sku": "SKU-001",
  "name": "iPhone 15",
  "spec": "黑色 256G",
  "createdAt": "2026-06-20T10:00:00.000Z",
  "updatedAt": "2026-06-20T10:00:00.000Z"
}
```

GET /products 返回 `data`：

```json
{
  "list": [/* Product 对象数组 */],
  "total": 56,
  "page": 1,
  "pageSize": 20
}
```

**错误码：**

| 接口 | code | 说明 |
|------|------|------|
| POST /products | 400 | 参数缺失 或 sku 已存在 |
| PUT /products/:sku | 404 | 商品不存在 |
| GET /products/:sku | 404 | 商品不存在 |

## 入库 / 出库 / 退货

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/api/inbound` | ALL | 采购入库（SKU + 仓库编码 + 数量） |
| POST | `/api/outbound` | ALL | 销售出库（SKU + 仓库编码 + 数量） |
| POST | `/api/return` | ALL | 退货（SKU + 仓库编码 + 数量 + type + remark） |

> 需求原文："仓库人员仅可进行入库、出库、库存查询操作"，因此对所有角色开放。

### 通用参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| bizNo | string | 是 | 业务单号（UUID），用于幂等性保证 |

### bizNo 幂等性设计

**作用层级：**

| 防护层级 | 问题 | 解决方案 |
|---------|------|----------|
| UI 层 | 用户双击提交 | 按钮锁（`disabled`） |
| 网络层 | HTTP 客户端自动重试（axios retry、超时重发） | bizNo（@unique 约束） |

**工作原理：**

```
客户端 → 生成 UUID: "abc-123" → 发请求（bizNo="abc-123"）
服务端 → 插入 bizNo="abc-123" → 成功
客户端 ← 网络超时，没收到响应
客户端 → axios 自动重试（bizNo="abc-123"）
服务端 → 插入 bizNo="abc-123" → @unique 报错 P2002 → 拒绝重复
```

bizNo 由客户端在打开表单时生成，绑定到一次操作。网络层的自动重试会携带相同的 bizNo，服务端通过 `@unique` 约束拒绝重复请求。

**局限性：**

| 场景 | 按钮锁 | bizNo |
|------|--------|-------|
| 双击提交 | ✅ 阻止 | ✅ @unique 拒绝 |
| 网络超时，自动重试 | ❌ 按钮已解锁 | ✅ 同一 UUID，拒绝 |
| 用户刷新页面，重新提交 | ❌ 锁消失 | ❌ 新 UUID，照样重复 |

bizNo 无法解决"用户主动刷新页面后重新提交"的问题，因为刷新会生成新的 UUID。这属于业务层去重问题，需通过其他手段处理（如业务规则检查、前端确认弹窗等）。

### 请求参数

| 接口 | 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| POST /inbound | bizNo | string | 是 | 业务单号（UUID） |
| | sku | string | 是 | 商品编码 |
| | warehouseCode | string | 是 | 仓库编码 |
| | quantity | number | 是 | 数量（> 0） |
| POST /outbound | bizNo | string | 是 | 业务单号（UUID） |
| | sku | string | 是 | 商品编码 |
| | warehouseCode | string | 是 | 仓库编码 |
| | quantity | number | 是 | 数量（> 0，≤ 当前库存） |
| POST /return | bizNo | string | 是 | 业务单号（UUID） |
| | sku | string | 是 | 商品编码 |
| | warehouseCode | string | 是 | 仓库编码 |
| | quantity | number | 是 | 数量（> 0） |
| | type | string | 是 | `RETURN_IN`（退货入库）/ `RETURN_OUT`（退货出库） |
| | remark | string | 是 | 备注（退货原因） |

### 响应示例

POST /inbound、/outbound、/return 返回 `data`：

```json
{
  "id": 1,
  "bizNo": "abc-123-def",
  "sku": "SKU-001",
  "warehouseCode": "WH-001",
  "type": "IN",
  "quantity": 100,
  "operatorId": 2,
  "remark": null,
  "createdAt": "2026-06-20T10:00:00.000Z"
}
```

**幂等性响应**（bizNo 已存在时）：

返回已处理的流水记录，`message` 为 `"已处理（重复请求）"`。

### 错误码

| 接口 | code | 说明 |
|------|------|------|
| POST /inbound | 400 | 参数缺失 / 数量无效 |
| POST /inbound | 404 | 商品 SKU 或仓库编码不存在 |
| POST /outbound | 400 | 参数缺失 / 数量无效 / 库存不足 |
| POST /outbound | 404 | 商品 SKU 或仓库编码不存在 |
| POST /return | 400 | 参数缺失 / type 无效 / 库存不足（RETURN_OUT 时） |
| POST /return | 404 | 商品 SKU 或仓库编码不存在 |

## 库存查询 / 流水查询

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/api/inventory` | ALL | 统一库存查询（根据参数决定返回格式） |
| GET | `/api/transactions` | ALL | 流水查询（按商品/仓库/时间/类型筛选） |

### GET /inventory 参数与返回

| 参数 | 返回格式 | 用途 |
|------|---------|------|
| `?sku=X` | 聚合视图（该商品 + totalQuantity + warehouses[]） | 查某商品在各仓库的分布 |
| `?warehouseCode=X` | 扁平列表 + 分页 | 查某仓库的所有商品库存 |
| 无筛选 | 扁平列表 + 分页 | 全量库存浏览 |

**聚合模式响应**（`?sku=SKU-001`）：

```json
{
  "code": 0,
  "data": {
    "sku": "SKU-001",
    "name": "iPhone 15",
    "spec": "黑色 256G",
    "totalQuantity": 150,
    "warehouses": [
      { "warehouseCode": "WH-001", "name": "主仓库", "quantity": 100 },
      { "warehouseCode": "WH-002", "name": "备用仓库", "quantity": 50 }
    ]
  }
}
```

**列表模式响应**（`?warehouseCode=WH-001` 或无筛选）：

```json
{
  "code": 0,
  "data": {
    "list": [
      {
        "id": 1,
        "sku": "SKU-001",
        "warehouseCode": "WH-001",
        "quantity": 100,
        "product": { "sku": "SKU-001", "name": "iPhone 15", "spec": "黑色 256G", "createdAt": "...", "updatedAt": "..." },
        "warehouse": { "code": "WH-001", "name": "主仓库", "address": "...", "createdAt": "...", "updatedAt": "..." }
      }
    ],
    "total": 10,
    "page": 1,
    "pageSize": 20
  }
}
```
### 架构演变记录

#### V1（初始设计）

四个接口分散在不同位置，都能查“商品 + 仓库 + 数量”：

- `GET /products/:sku` → include inventory + warehouse
- `GET /warehouses/:code` → include inventory + product
- `GET /inventory` → 扁平列表 + 分页
- `GET /inventory/summary` → 按商品聚合

**问题**：功能重复，product 详情和 inventory 都能查同一份数据，前端不确定该调哪个。

#### V2（当前版本）

职责分离，库存统一走 `/inventory`：

- `GET /products/:sku` → 纯主数据（sku, name, spec）
- `GET /warehouses/:code` → 纯主数据（code, name, address）
- `GET /inventory?sku=X` → 聚合视图（totalQuantity + warehouses[]）
- `GET /inventory?warehouseCode=X` → 扁平列表（该仓库的所有商品库存）
- `GET /inventory` → 扁平列表（全量浏览）

**收益**：
- 接口职责单一：商品/仓库详情只返回主数据，库存查询统一入口
- 消除重复：不再有 4 个接口查同一张表
- 前端调用明确：要库存信息就走 `/inventory`
