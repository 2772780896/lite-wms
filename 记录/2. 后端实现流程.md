## Task 1：技术选型与设计原则

| 层面 | 选型 | 选择原因 |
|------|------|----------|
| 后端框架 | Node.js + Express | 轻量、生态成熟、适合中小型项目 |
| ORM | Prisma | 类型安全、Schema 清晰、内置事务支持 |
| 数据库 | SQLite（开发）/ PostgreSQL（生产） | SQLite 零配置快速开发，生产环境可迁移 |
| 前端 | React + Tailwind CSS | 轻量、不依赖组件库（不要求完整实现） |
| 认证 | JWT | 无状态，适合 RESTful API |

**核心设计原则：**

- **业务解耦**：商品、仓库、库存、流水分离，避免数据冗余。
- **并发安全**：摒弃"先查再改"，采用数据库层面的原子更新（`updateMany` + `WHERE quantity >= count`），防止超卖。
- **幂等性**：流水表 `bizNo` 唯一索引，防止前端重复提交。
- **可追溯性**：所有出入库操作必须留存流水记录。

## Task 4：认证与权限

### 认证流程

### JWT 认证原理

JWT 是**自包含**的：签发时把 `userId` 和 `role` 写入 payload 并用服务端密钥（SECRET_KEY）生成签名，验证时用同一把密钥校验签名是否合法，签名通过即可直接解出 payload 中的 userId 和 role，**不需要查数据库**。

- 攻击者可以读取 payload（base64 明文），但无法伪造签名（没有密钥）。
- Payload 中不要放敏感信息（如密码）。

```
登录：POST /api/auth/login  { username, password }
  → 数据库查出用户 → bcrypt.compare 验证密码（仅登录时调用一次，~100ms）
  → jwt.sign({ userId, role }, SECRET_KEY)  // 签发，payload 携带角色
  → 返回 { token, role } 给前端（role 明文返回，供前端控制 UI）

后续请求：Header: Authorization: Bearer <token>
  → jwt.verify(token, SECRET_KEY)  // 验签 + 解出 payload
  → 签名合法 → 得到 { userId, role }  // 不需要查数据库
  → 签名被篡改 → 验证失败，返回 401
```

### 中间件

- **auth.js（身份解析）**：从 Header 取出 token → `jwt.verify` 验签 → 将解出的 `{ userId, role }` 挂载到 `req.user`。挂载的目的是让后续的路由处理函数可以直接通过 `req.user.userId` 获取操作人 ID（用于记录流水的 operatorId），不需要每个处理函数都重复解析一次 token。所有需登录路由都过此中间件。
- **rbac.js（路由守卫）**：本质上就是后端的路由守卫——给每个路由标注"谁能访问"。接收允许的角色列表，检查 `req.user.role` 是否在列表中，不在则返回 403。与前端路由守卫的关系：前端守卫控制 UI 展示（体验层），rbac 控制接口访问（安全层），两者各管各的，后端 rbac 才是真正的安全防线。

## Task 6：统一响应格式与错误处理

> 当前未抽取独立的 `utils/response.js` 工具函数，响应格式在各 controller 中内联保持一致。

```javascript
// 成功
{ "code": 0, "data": { ... }, "message": "ok" }

// 失败
{ "code": 40001, "data": null, "message": "库存不足" }
```

在 `app.js` 中设置全局错误捕获中间件，将业务层 `throw new Error()` 统一转为 JSON 响应。

## 高并发应对策略总结

| 场景 | 解决方案 | 说明 |
|------|----------|------|
| 超卖问题 | 原子更新 | `updateMany` + `quantity: { gte: count }`，杜绝"先查再改"竞态 |
| 重复提交 | 业务单号幂等 | `bizNo` 唯一索引，数据库拦截重复插入 |
| 性能瓶颈 | 批量操作 | 批量入库在事务中循环处理；非核心逻辑可放消息队列（未来扩展） |
| 数据一致性 | 数据库事务 | `$transaction` 确保库存扣减和流水记录同成同败 |
