# WMS Lite 业务操作汇总

## 1. 认证

- **登录**：用户输入用户名和密码，系统验证后签发 JWT（含 userId、role），前端存储 Token 用于后续请求。

> **→ 需要：用户表**
> - 用户名（唯一）、密码（哈希）、显示名称、角色（管理员/仓库人员）

---

## 2. 商品管理（仅管理员）

- **新增商品**：输入 SKU 编码、商品名称、规格（可选，用于区分同类商品的不同属性，如"500ml"、"红色/XL码"、"304不锈钢"——同一商品名可能因规格不同而对应不同 SKU，仓库人员需要通过规格快速识别具体是哪个版本的商品），系统创建商品记录。
- **编辑商品**：根据商品 ID 修改名称、规格等信息。
- **查询商品列表**：支持按名称或 SKU 模糊搜索，返回商品列表。
- **查询商品详情**：根据商品 ID 返回单个商品信息。

> **→ 需要：商品表**
> - 商品编码 SKU（主键，扫码/输入直接定位商品）、名称、规格（可选）

---

## 3. 仓库管理（仅管理员可增改，所有人可查）

- **新增仓库**：输入仓库编码、名称、地址（可选），系统创建仓库记录。
- **编辑仓库**：根据仓库 ID 修改名称、地址等信息。
- **查询仓库列表**：返回所有仓库信息。
- **查询仓库详情**：根据仓库 ID 返回单个仓库信息。

> **→ 需要：仓库表**
> - 仓库编码（主键，扫码/输入直接定位仓库）、名称、地址（可选）

---

## 4. 采购入库（管理员 + 仓库人员）

- **单品入库**：操作员输入/扫描商品 SKU + 仓库编码 + 数量，系统在事务中执行"库存增加 + 记录流水"。如果该商品在该仓库还没有库存记录，则自动创建（通过库存表的 `@@unique([sku, warehouseCode])` 联合唯一约束 + Prisma 的 `upsert` 实现：数据库层面单条 SQL 完成"有则加、无则建"，不存在"先查再改"的竞态条件，避免并发入库时创建出重复的库存记录）。
- **批量入库**：一次提交多个商品的入库信息，在同一个事务中批量处理。

> **→ 需要：库存表**
> - 商品 sku + 仓库编码（联合唯一）、可用数量
>
> **→ 需要：流水表**（入库/出库/退货共用）
> - 业务单号（唯一，幂等性）、商品 sku、仓库编码、类型（入库/出库/退货入库/退货出库）、数量、操作人 ID、备注（退货原因等）、操作时间
>
> **→ 关联**：sku 和仓库编码直接作为外键引用商品表和仓库表

---

## 5. 销售出库（管理员 + 仓库人员）

- **单品出库**：操作员输入/扫描商品 SKU + 仓库编码 + 数量，系统原子校验库存是否充足，在事务中执行"库存扣减 + 记录流水"。库存不足则拒绝操作。

> **→ 复用：库存表**（原子扣减，WHERE quantity >= count）
> **→ 复用：流水表**（记一笔 OUT 类型记录）
>
> **关键约束**：库存表的 quantity 字段需要支持数据库层面的原子更新（updateMany + gte 条件）

---

## 6. 退货（管理员 + 仓库人员）

- **销售退货（退货入库）**：客户退回商品，操作员输入 SKU + 仓库编码 + 数量 + 退货原因，库存增加，流水类型标记为 RETURN_IN，备注退货原因。
- **采购退货（退货出库）**：退回供应商，操作员输入 SKU + 仓库编码 + 数量 + 退货原因，库存扣减，流水类型标记为 RETURN_OUT，备注退货原因。

> **→ 复用：库存表**（增或减）
> **→ 复用：流水表**（类型为 RETURN_IN / RETURN_OUT，备注字段记录退货原因）
>
> **补充**：流水表的"备注"字段在退货场景下为必填

---

## 7. 库存查询（管理员 + 仓库人员）

- **按仓库查库存**：查看某个仓库下所有商品及数量。
- **按商品查库存**：查看某个商品在各仓库的库存分布。
- **库存汇总**：按商品汇总总库存（跨仓库求和），用于管理员看板。

> **→ 复用：库存表**，需要关联查询商品表（名称、SKU）和仓库表（名称、编码）
>
> **查询需求**：支持按 warehouseCode 筛选、按 sku 筛选、按 sku 分组求和

---

## 8. 流水查询（管理员 + 仓库人员）

- **查询流水记录**：支持按商品、仓库、时间范围、流水类型筛选，返回出入库操作历史。

> **→ 复用：流水表**
>
> **查询需求**：支持按 sku、warehouseCode、type、createdAt 范围筛选
>
> **索引需求**：商品+仓库 组合索引、时间索引

---

## 管理粒度说明：数量级管理 vs 单品级管理

在仓储系统中，"编码入库出库"存在两种不同的管理粒度：

- **数量级管理（本项目采用）**：一个 SKU = 一种商品，库存只记录数量，不区分每一件实物。适用于快消品、标品。
- **单品级管理（本项目不采用）**：每一件实物有唯一序列号，入库/出库精确到具体某一件。适用于高价值商品。

本项目选择数量级管理的理由：需求匹配标品仓储场景、复杂度可控、遵循 YAGNI 原则。

> 关于两种粒度的详细对比、并发防超卖方案、以及未来单品级扩展的三种架构方案，详见 `2. 管理粒度与并发方案说明.md`。

## 推导总结

从以上业务操作逐步推导，共需要 **5 张表**：

| 表 | 推导来源 | 核心字段 |
|----|----------|----------|
| **用户表** | 认证 | 用户名、密码、名称、角色 |
| **商品表** | 商品管理 | SKU（主键）、名称、规格 |
| **仓库表** | 仓库管理 | 编码（主键）、名称、地址 |
| **库存表** | 入库、出库、退货、库存查询 | sku + 仓库编码（联合唯一）、数量 |
| **流水表** | 入库、出库、退货、流水查询 | 业务单号（唯一）、sku、仓库编码、类型、数量、操作人ID、备注、时间 |
