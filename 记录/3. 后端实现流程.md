## Task 1：技术选型与设计原则

| 层面 | 选型 | 选择原因 |
|------|------|----------|
| 后端框架 | Node.js + Express | 轻量、生态成熟、适合中小型项目 |
| ORM | Prisma | 类型安全、Schema 清晰、内置事务支持 |
| 数据库 | SQLite（开发）/ PostgreSQL（生产） | SQLite 零配置快速开发，生产环境可迁移 |
| 前端 | React + Ant Design | 管理后台可视化（不要求完整实现） |
| 认证 | JWT | 无状态，适合 RESTful API |

**核心设计原则：**

- **业务解耦**：商品、仓库、库存、流水分离，避免数据冗余。
- **并发安全**：摒弃"先查再改"，采用数据库层面的原子更新（`updateMany` + `WHERE quantity >= count`），防止超卖。
- **幂等性**：流水表 `bizNo` 唯一索引，防止前端重复提交。
- **可追溯性**：所有出入库操作必须留存流水记录。

## Task 4：认证与权限

### 认证流程

### JWT 认证原理

JWT 是**自包含**的：签发时把 `userId` 和 `role` 写入 payload 并用服务端密钥（SECRET_KEY）生成签名，验证时用同一把密钥校验签名是否合法，签名通过即可直接解出 payload 中的 userId 和 role，**不需要查数据库**。

- 攻击者可以读取 payload（base64 明文），但无法伪造签名（没有密钥）。
- Payload 中不要放敏感信息（如密码）。

```
登录：POST /api/auth/login  { username, password }
  → 数据库查出用户 → bcrypt.compare 验证密码（仅登录时调用一次，~100ms）
  → jwt.sign({ userId, role }, SECRET_KEY)  // 签发，payload 携带角色
  → 返回 { token, role } 给前端（role 明文返回，供前端控制 UI）

后续请求：Header: Authorization: Bearer <token>
  → jwt.verify(token, SECRET_KEY)  // 验签 + 解出 payload
  → 签名合法 → 得到 { userId, role }  // 不需要查数据库
  → 签名被篡改 → 验证失败，返回 401
```

### 中间件

- **auth.js（身份解析）**：从 Header 取出 token → `jwt.verify` 验签 → 将解出的 `{ userId, role }` 挂载到 `req.user`。挂载的目的是让后续的路由处理函数可以直接通过 `req.user.userId` 获取操作人 ID（用于记录流水的 operatorId），不需要每个处理函数都重复解析一次 token。所有需登录路由都过此中间件。
- **rbac.js（路由守卫）**：本质上就是后端的路由守卫——给每个路由标注"谁能访问"。接收允许的角色列表，检查 `req.user.role` 是否在列表中，不在则返回 403。与前端路由守卫的关系：前端守卫控制 UI 展示（体验层），rbac 控制接口访问（安全层），两者各管各的，后端 rbac 才是真正的安全防线。

## Task 7：核心业务逻辑实现

### 7.1 采购入库

```javascript
async function handleInbound(sku, warehouseCode, count, bizNo, operatorId) {
  return await prisma.$transaction(async (tx) => {
    // 1. 编码反查（编码入库的核心：不需要在下拉菜单里选，直接扫码/输入编码）
    const product = await tx.product.findUnique({ where: { sku } });
    const warehouse = await tx.warehouse.findUnique({ where: { code: warehouseCode } });
    if (!product || !warehouse) throw new Error('商品或仓库编码不存在');

    // 2. 增加库存（upsert：有则加，无则建）
    await tx.inventory.upsert({
      where: { productId_warehouseId: { productId: product.id, warehouseId: warehouse.id } },
      create: { productId: product.id, warehouseId: warehouse.id, quantity: count },
      update: { quantity: { increment: count } },
    });

    // 3. 记录流水
    await tx.transactionLog.create({
      data: { bizNo, productId: product.id, warehouseId: warehouse.id, type: 'IN', quantity: count, operatorId },
    });

    return { success: true, message: '入库成功' };
  });
}
```

**关于批量入库（当前不实现）：**

当前只做单品入库（扫一个 → 确认 → 下一个），不实现批量入库。原因：WMS 扫码场景的工作流是逐件操作，不需要一次性提交多个商品。

如果未来需要批量入库，有两条改进方向：

| 方向 | 做法 | 优劣 |
|------|------|------|
| **JS 层循环** | 在事务内 `for...of` 逐条执行 upsert + create流水 | 简单，但 N 件商品 = N 次 SQL 网络往返，规模上百时慢 |
| **数据库层批量** | 用原生 SQL `INSERT ... ON CONFLICT UPDATE` 一条语句包含所有行 | 快（1 次网络往返），但需要手写 SQL，且流水记录仍需单独插入 |

两者都需要在事务中保证"库存更新 + 流水记录"的原子性（`bizNo` 唯一约束可能因重复提交导致流水插入失败，此时需回滚库存）。

### 7.2 销售出库

```javascript
async function handleOutbound(sku, warehouseCode, count, bizNo, operatorId) {
  return await prisma.$transaction(async (tx) => {
    // 1. 编码反查
    const product = await tx.product.findUnique({ where: { sku } });
    const warehouse = await tx.warehouse.findUnique({ where: { code: warehouseCode } });
    if (!product || !warehouse) throw new Error('商品或仓库编码不存在');

    // 2. 原子扣减（WHERE 边界条件防超卖）
    const result = await tx.inventory.updateMany({
      where: { productId: product.id, warehouseId: warehouse.id, quantity: { gte: count } },
      data: { quantity: { decrement: count } },
    });
    if (result.count === 0) throw new Error('库存不足或商品不存在');

    // 3. 记录流水
    await tx.transactionLog.create({
      data: { bizNo, productId: product.id, warehouseId: warehouse.id, type: 'OUT', quantity: count, operatorId },
    });

    return { success: true, message: '出库成功' };
  });
}
```

### 7.3 退货

退货是出入库的逆操作，核心逻辑相同，区别在于 `type` 和 `remark`：

- **销售退货（退货入库）**：库存增加，`type = 'RETURN_IN'`，`remark` 必填（退货原因）。
- **采购退货（退货出库）**：库存扣减，`type = 'RETURN_OUT'`，`remark` 必填（退货原因）。

```javascript
// 销售退货示例（逻辑同入库，只是 type 不同）
await tx.transactionLog.create({
  data: { bizNo, productId, warehouseId, type: 'RETURN_IN', quantity: count, operatorId, remark: '客户退货' },
});
```

## 高并发应对策略总结

| 场景 | 解决方案 | 说明 |
|------|----------|------|
| 超卖问题 | 原子更新 | `updateMany` + `quantity: { gte: count }`，杜绝"先查再改"竞态 |
| 重复提交 | 业务单号幂等 | `bizNo` 唯一索引，数据库拦截重复插入 |
| 性能瓶颈 | 批量操作 | 批量入库在事务中循环处理；非核心逻辑可放消息队列（未来扩展） |
| 数据一致性 | 数据库事务 | `$transaction` 确保库存扣减和流水记录同成同败 |

> **为什么不用 Redis/MQ？** 当前业务体量（中小企业），数据库事务足以支撑，避免过度设计。


建议你先从简单的开始：认证 → 商品 CRUD → 仓库 CRUD → 入库 → 出库 → 退货 → 库存查询 → 流水查询，按这个顺序逐个写，每写完一个就能测一个。
