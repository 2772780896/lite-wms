分析需求，并确定所需数据表与操作流程。具体分析见 "记录/1. 需求分析与数据表构建.md" ，简要说明如下：
    
    1. 用户登录与角色认证 => 用户表，包含用户基本信息，同时以角色标明权限，以bcrypt密文存储密码 => 详见 "backend/prisma/schema.prisma"
                           登录系统，通过异步bcrybt计算密码并验证，以用户id和角色作为jwt的payload签发token => 详见 "backend/controllers/login.js"
                           第一层验证，以jwt解析token验证token，并得到用户id和角色，挂载到请求中 => 详见 "backend/middleware/auth.js"
                           第二层验证，相当于路由守卫，通过手动设置当前路由可访问的角色，来拦截请求中不满足条件的用户访问 => 详见 "backend/middleware/rbac.js"
    
    2. 商品与仓库的基本管理 => sku为主键的商品表，可确保全流程以sku为数据进行通信
                             code为主键的仓库表
                             基本新增、编辑、查询的接口，但此处为了不与之后的库存查询冲突，仅实现对表中原有字段的查询，不进行关联查询 => 详见 "backend/controllers/product.js" & "backend/controllers/warehouse.js"
    
    3. 入库与退货入库 => 流水表，对用户、商品、仓库表建立多对一的关系，以此来记录操作
                        库存表，对商品、仓库表建立多对一的关系，并通过同时约束商品对仓库之间的唯一性来确保数据的更新和统计更直观
                        入库流程，入库时先增加库存表中对应商品的数量，再通过事务锁确保流水表写入时，如果bizNo的唯一性避免因网络重试而导致重复写入，整个事务可回滚 => 详见 "backend/controllers/transaction.js"
    
    4. 出库与退货出库 => 出库流程，将有边界条件的查询与对库存表的更新，整合到一条数据库操作语句中，实现原子化以避免延时导致的超量出库，同时用事务锁保障流水表的写入 => 详见 "backend/controllers/warehouse.js"
    
    5. 库存查询 => 全量查询，返回库存表中所有记录，直观展示每个商品在每个仓库的数量，同时关联商品、仓库表中对应的信息 => 详见 "backend/controllers/invenroty.js"
                  单商品查询，返回单个商品在各个仓库中的数量，以及总数，总数通过循环累加各仓库数实现，若计数量大，可将计算任务推给数据库层减小主线程负担
                  单仓库查询，类似全量查询，不对总数进行汇总
                  排序，针对全量查询数据过多的问题，封装排序功能，接收请求中排序字段与顺序，并手动设置数据表可使用排序字段，直接输出可插入orm语句中的排序需求 => 详见 "backend/lib/sort.js"


补充回答：
    
    一. 管理的最小单位被设置为商品的sku层，为什么不设置成sn层，以及如果要实现sn层该怎么做？
        具体分析见 "想法/3. 管理颗粒度.md" ，简要说明如下：
        1. sn层意味着更精细的数据表结构、更复杂的统合入库、跨度更大的总数统计，目前先实现sku级管理，并预留向sn级发展的方向。
        2. 如果要实现sn级，需要先构建单品表，并用status字段明确标明每一个体的状态，设置更精准的边界条件
        3. 批量入库时才用统合传输的模式，js层不参与遍历每一个个体，而是直接将集合体交给数据库层去并发处理。同时为了避免部分失败导致的全体回滚的问题，采用反向查询开始列表中的成功个体，从而可以从总列表切出失败个体，这样的数据库操作检验模式。
        4. 考虑但单体频繁入库造成的总数字段频繁修改，以及为了确保一致性而必须采用事务锁或原子化来加大数据库执行开销，容易造成数据库阻塞的情况。总数查询不再跨多个表维护一个总数，而是采用创建索引实时查询的模式。并提出其他解决方案。
    
    二. js是单线程的，要怎么支撑多请求的后端，Node又是怎么工作的？
        具体分析见 "想法/1. JS与Node并发学习笔记.md" ，简要说明如下：
        1. js在主线程中确实是单线程工作的，但如果只是执行接收前端请求并执行同步流程，js是能撑得住多用户访问的，只要不在同步代码中写耗时计算或大循环
        2. 当需要耗时操作，如fs、multer、bcrybt组件时，会让Node去调用libuv线程进行处理，不阻塞主线程。或直接将异步操作交给数据库或操作系统等天生适合并发的场景去处理
        3. 当实在需要在js中执行复杂计算时，用Worker Threads真正多核打开另一个js来运行
    
    三. 多文件的并发怎么处理？
        具体分析见 "想法/2. 大文件解决方案.md" ，简要说明如下：
        1. 不让js去遍历执行异步，而是像前面说的打包交给可以并发的组件。
        2. 当文件太大时，为保护内存和js不被阻塞，考虑流式传输，将文件转为用连续空间存储的Buffer格式，使得对该储存空间的切片可以使用无需遍历的subarray指针，完成快速切片。并通过切片传输的方式确保线程不被阻塞。
